# Validated Finding

## Summary
- ID: F-001
- Title: Capability recommendation layer falsely reports unsupported one-sided PRF/Seed use contracts as supported
- Severity: low
- Confidence: high
- Root Cause Key: backend/recommend.rs:supports_mode:missing-coupled-use-contracts:false-capability-acceptance
- Disclosure tier: full internal

## Affected area
- Component: capability recommendation / mode selection truthfulness
- Path(s): src/backend/recommend.rs; src/model/core.rs; src/ops/enforcement.rs; cross-ref: src/ops.rs; src/model/capability.rs
- Function(s): `supports_mode`, `unsupported_mode_reason`, `report_supports_mode`, `UseCase::validate_for_mode`, `resolve_mode`
- Ownership if known: hardening/state-model refactor surface

## What was proven
- Trigger conditions: evaluate PRF/Seed support for `Verify`-only, `Decrypt`-only, or `Ssh`-only use sets.
- Proof method: executed repository unit tests that prove both sides of the contradiction.
- Observed result: `backend::recommend::tests::prf_support_matches_the_current_identity_surface` passes with one-sided PRF support claims, while `ops::enforcement::tests::setup_rejects_verify_only_use_contract`, `setup_rejects_decrypt_without_encrypt_use_contract`, and `setup_rejects_ssh_without_sign_use_contract` all pass and reject the same coupled-use shapes during setup.
- Oracle used: targeted `cargo test` runs plus direct code-path cross-reference.

## Root cause
- Explanation: `src/backend/recommend.rs` locally models allowable uses with `allowed_for_mode(...)` and SSH algorithm checks, but omits the canonical coupled-use requirements enforced in `src/model/core.rs` (`verify => sign`, `decrypt => encrypt`, `ssh => sign`).
- Why this is the likely cause: the contradiction disappears if the recommendation layer reuses the same coupled-use contract; no alternate state path explains why recommendation accepts requests that setup later refuses.
- Duplicate check result: unique in this workspace; no duplicate root-cause key found.

## Impact triage
- Boundary crossed: truthful capability-model boundary between preflight inspection/mode recommendation and actual identity creation enforcement.
- Reachability: reachable wherever `CapabilityProbe::supports_mode`, `report_supports_mode`, or `recommended_mode` results are trusted before setup.
- Preconditions: caller relies on capability report or explicit support check instead of only the final setup result.
- Breadth: affects PRF/Seed coupled-use requests in the recommendation layer; does not currently create a materialized invalid identity because setup still rejects.
- Why fix now: this branch is explicitly hardening state/model truthfulness. Leaving the drift in place preserves a false-positive capability surface that can mislead automation, tests, and future refactors.

## Fix handoff
- Likely fix direction: centralize the coupled-use contract in a shared helper and apply it inside `src/backend/recommend.rs` before a mode is reported as supported or recommended.
- Suggested regression test: add recommendation-layer tests proving PRF/Seed reject `Verify` without `Sign`, `Decrypt` without `Encrypt`, and `Ssh` without `Sign`.
- Open questions: whether `inspect` output schema should expose both raw primitive support and policy-valid identity surfaces separately.
