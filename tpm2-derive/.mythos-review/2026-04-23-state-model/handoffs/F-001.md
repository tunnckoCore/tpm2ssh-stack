## Why the fixing agent should care
- User-visible or security impact: preflight capability reporting is not truthful for some request shapes, so callers can be told a use-set is satisfiable and then fail later during setup.
- Affected component: capability recommendation / inspect / mode resolution surface.
- Trust boundary: model truthfulness boundary between recommendation and enforcement.

## What is already known
- Repro status: proven with existing unit tests and targeted cargo runs.
- Root cause status: high-confidence; duplicated policy logic in `backend/recommend.rs` omits coupled-use checks present in `model/core.rs`.
- Evidence quality: strong.

## Likely fix direction
- Checks to add or restore: require the same coupled-use contract in recommendation-layer support checks.
- State transitions to harden: do not recommend or mark a mode as supported unless the expanded use-set is also policy-valid as an identity surface.
- Bounds / lifetime / ownership rules to enforce: keep one canonical validator for use-coupling rules.
- Tests to add: recommendation-layer negative tests for verify-only, decrypt-only, ssh-only.

## Suggested first implementation steps
1. Extract a shared helper that validates expanded use vectors without duplicating mode-specific policy in two places.
2. Use that helper inside `supports_mode(...)` and `unsupported_mode_reason(...)` in `src/backend/recommend.rs`.
3. Add regression tests mirroring the existing enforcement tests at the recommendation layer.
