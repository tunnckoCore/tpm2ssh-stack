# Validated Finding

## Summary
- ID: 2026-04-23-sign-ssh-F001
- Title: `ssh-add` socket validation can be bypassed with an intermediate symlink and post-validation swap
- Severity: High
- Confidence: High
- Root Cause Key: `ssh:add:resolve_socket:intermediate-symlink-socket-redirection`
- Disclosure tier: full internal

## Affected area
- Component: SSH agent export / socket validation
- Path(s): `src/ops/ssh.rs:159-207`, `src/ops/ssh.rs:401-476`
- Function(s): `add_derived_identity`, `resolve_socket`
- Ownership if known: CLI ops / SSH export path

## What was proven
- Trigger conditions: a caller supplies `--socket` or `SSH_AUTH_SOCK` that resolves through a symlinked ancestor directory; the symlink target changes after validation but before the client consumes the path.
- Proof method: custom local repro harness using the public `add_with_backend` API, a fake seed backend, and a deriver that swaps the ancestor symlink between `resolve_socket()` and `SshAddClient::add_private_key()`.
- Observed result: the operation accepted `/tmp/.../linkdir/agent.sock`, returned that same noncanonical path in `SshAddResult`, and the client observed that `canonicalize(socket)` now resolved to `/tmp/.../attacker/agent.sock` (`swapped_to_attacker=true`).
- Oracle used: `./.mythos-review/2026-04-23-sign-ssh/evidence/ssh-symlink-race/repro.rs`, `./.mythos-review/2026-04-23-sign-ssh/evidence/ssh-symlink-race/output.txt`

## Root cause
- Explanation: `resolve_socket()` only rejects a symlink on the final socket pathname via `fs::symlink_metadata(&path)` and checks the parent with `fs::metadata(parent)`, which follows symlinks instead of rejecting them. `add_derived_identity()` then validates the socket once, derives the private key, and finally passes the original raw `PathBuf` to `client.add_private_key()`.
- Why this is the likely cause: the repro succeeds only because ancestor symlinks are accepted and the noncanonical path is reused after the swap; once the client canonicalizes the path, it lands on the attacker socket, matching the code path exactly.
- Duplicate check result: no duplicate in the scoped segment; unique root cause.

## Impact triage
- Boundary crossed: TPM/derived private key material can be sent to the wrong local IPC peer despite the tool's socket trust checks.
- Reachability: reachable in every `ssh-add` flow for PRF/seed identities once the operator intentionally enables secret egress.
- Preconditions: attacker can influence the socket pathname through an ancestor symlink and win a swap/race before `ssh-add` consumes the path.
- Breadth: affects both explicit `--socket` and inherited `SSH_AUTH_SOCK` because both funnel through `resolve_socket()`.
- Why fix now: this is the last gate before private key material leaves TPM-only protection; a destination-confusion bug here nullifies the operator's intent and can exfiltrate a long-lived SSH key to an attacker-controlled agent.

## Fix handoff
- Likely fix direction: reject symlinks in every socket path component (not just the leaf), canonicalize and pin the verified socket path before any expensive work, and use the verified canonical path for the later `ssh-add` invocation.
- Suggested regression test: add a unit/integration test proving `resolve_socket()` rejects ancestor symlinks and another proving a swapped ancestor cannot redirect the path observed by the client.
- Open questions: whether to permit any symlinked ancestors at all, and whether a stronger `openat2(..., RESOLVE_NO_SYMLINKS)`-style implementation is desired on Linux for future-proofing.
